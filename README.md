# AI FlakeTester: Output Reliability Checker

A local LLM benchmarking tool that validates whether a model outputs valid JSON or hallucinates.

## Project Overview

When an AI model is asked to output valid JSON schemas, it will often hallucinate and include 
artefacts within the output (such as markdown formatting or incorrect types). If generating 
large datasets, this poses a significant risk — poisoned data used for training can result in 
a vastly inferior model output.

This script validates a model's JSON output reliability and assigns it a flake score based on 
hallucinated outputs. A score of 0 means the model made no mistakes. The 
script also uses Ollama telemetry to calculate average tokens/sec generation speed.

## Tech Stack

### Python
Coded entirely in Python. Uses asyncio for concurrent prompt dispatch, Pydantic V2 for strict 
JSON validation, and argparse for the CLI.

### Ollama API
Used to query locally installed models. The `AsyncClient` is shared across all concurrent tasks 
for efficient connection management.

### Pydantic V2
`TypeAdapter.validate_json()` with strict mode is used to validate model outputs against 
predefined schemas. Schemas are defined per difficulty level in `DIFFICULTY_CONFIG`.

### Asyncio
Prompts are dispatched concurrently using `asyncio.gather`. Failed tasks are caught via 
`return_exceptions=True` and counted as flakes rather than crashing the run.

### Pytest
Full async test suite using `pytest` and `pytest-asyncio`. Uses `unittest.mock` with `AsyncMock` 
to isolate network calls and filesystem operations for deterministic testing.

### Psutil
Used to gather memory usage at the peak of the process to show how much data was required
during the testing procedure.

### Pandas
The data from `models_info.csv` is read and converted into a dataframe the columns of which
are later used to create graphs to display the primary measurement metrics using a bar graph.
It also drops any rows if they are not required by the Latency graph if the total runs was less than
the minimum of 100 so they don't affect the data representation.

### Plotly
Based on the dataframe generated by using pandas plotly creates three graphs in `results.html`
The first graph shows the `Flake Score`, the second the `Avg. T/s` and the third compares the `P95`
`and P99 Latencies`. Each graph is compared by model, difficulty, and total runs.

### Pypdf
Used in the RAG benchmarking script to extract chunks from the desired pdf easiy.

### Nomic-embed-test
Used as a small model to embed the prompt using `ollama.embed`. It size allows us to quickly embed the extracted text.

### FAISS
Stores the embedded document chunks and retrieves the most semantically relevant one at query time using nearest-neighbour search.

## Prerequisites

### Ollama Setup

**Linux:** Ollama does not run as a GUI. Start the server manually in a terminal before 
running the script and keep it running:
```
ollama serve
```

**Windows/macOS:** The Ollama desktop application handles this automatically.

Pull the desired model before running:
```
ollama pull qwen3:1.7b
```
Pull the embedding model before running the RAG benchmark:
```
ollama pull nomic-embed-text:v1.5
```

### Python Dependencies

Python 3.10 or higher required. Install dependencies:
```
pip install -r requirements.txt
```

## Usage
```
python main.py --model qwen3:1.7b --run 100 --difficulty hard --concurrency 10
```
```
python rag_benchmark.py --model qwen3:1.7b --run 100 --difficulty hard --concurrency 10 --pdf rag_benchmark.pdf
```
```
python visualise.py
```

## Evaluation Metrics

**Flake Score:** Total number of outputs that failed strict Pydantic V2 validation. Catches 
missing keys, type coercion errors (e.g. `"90"` instead of `90`), and markdown wrapping. 
Network failures are also counted as flakes.

**Average T/s:** Model generation speed calculated from Ollama's `eval_count` (tokens) and 
`eval_duration` (nanoseconds). Tasks with zero duration are excluded to prevent division errors.

**P95/P99 Latency:** Percentile latency calculated from individual prompt response times in seconds. P95 means 95% of requests completed faster than this value. Requires a minimum of 100 runs — reported as N/A otherwise.

**Memory Usage (MB):** Peak resident set size of the process sampled immediately after all concurrent tasks complete, representing the highest memory footprint during the benchmark run.

**CSV Logging:** Results are appended to `models_info.csv` with columns: Model Name, Total Runs, Flake 
Score, Avg. T/s, Test Difficulty, P95 Latency, P99 Latency, Memory Usage (MB).

## RAG Benchmark
The RAG pipeline chunks a provided PDF by section, embeds each chunk using `nomic-embed-text:v1.5`, and retrieves the most relevant chunk at query time to inject as context into the difficulty prompt.  Results are written to `models_info.csv` with a `(RAG)` suffix for direct comparison in `visualise.py`.

On hard difficulty, RAG had negligible impact on gemma4:e4b — flake score moved from 34 to 35 across 100 runs, suggesting structured output capability is the limiting factor rather than context availability. 
On ultra the effect was significant, with flake score dropping from roughly 80 to 15 across repeated runs. This indicates that injecting relevant context meaningfully improves reasoning on deeply nested schemas with complex enum inference requirements. A minimum of 100 runs is recommended for reliable RAG vs non-RAG comparison.

## Difficulty Tiers
| Difficulty | Target Models | Description |
|------------|---------------|-------------|
| `easy` | Sub-3B parameter models | Flat schema, explicit values, no inference required |
| `med` | 3B–10B parameter models | Single nesting level, mild ambiguity in source text |
| `hard` | 30B+ models | Two nesting levels, enum inference, nullable fields |
| `ultra` | Frontier models only  | Three nesting levels, multiple enums requiring clinical inference, asymmetric nullable fields |
